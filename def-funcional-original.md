## 1. Purpose and Context

Memo is an organizational intelligence system for development teams using AI agents. It preserves and makes queryable the knowledge generated during software development — architectural decisions, technical rationale, integration contracts, discarded alternatives — at both the individual repository and full ecosystem scale.

The core use case has two complementary dimensions:

**Dimension 1 — Current repo memory:** the agent remembers what was decided in previous stories of the same repo and why, avoiding repeated debates and maintaining architectural consistency.

**Dimension 2 — Ecosystem intelligence:** the agent can query how another product is built, what another team decided on a topic, and where the right integration point is — without leaving its workflow.

### Problem it solves

AI agents have no memory between sessions and no visibility into the ecosystem. Every time an agent starts a task it begins from scratch. In platforms with many repositories this causes:

- Repeated debates already resolved, within and across teams
- Architectural inconsistencies between stories and between repos
- Time lost integrating with other products due to lack of context
- Knowledge loss when developers rotate
- Inability of agents to learn from the system's history

### What Memo is NOT

- Not a task manager or story tracker (Beads handles that)
- Not a replacement for formal documentation or wikis
- Not a chat tool or conversational assistant
- Not a replacement for version control (git)
- Not session memory between IDEs (Memorix handles that)

### Differentiation from similar tools

| Tool | What it solves | Gap Memo covers |
|---|---|---|
| Memorix | Session continuity per agent, per machine | No cross-repo or cross-team visibility |
| Mem0 | Conversational memory with LLM compression | No structured schema or repo relationships |
| Beads | Task planning and execution with dependencies | Does not capture rationale or architectural knowledge |
| ADR tools | Markdown decision documents in the repo | No semantic search or cross-repo agent access |

## 2. System Architecture

### Technology stack

| Component | Technology | Rationale |
|---|---|---|
| Vector Store | Qdrant Cloud (free tier) | Native semantic search + compound exact filters in the same query. Docker for local, cloud for multi-team. No code changes between both. |
| Embeddings | Configurable · OpenAI `text-embedding-3-small` recommended | Swappable via adapter. $0.02/M tokens. Estimated total cost <$1 USD per 10K entries in 12 months. |
| CLI | TypeScript + pnpm | Consistent with existing project stack. Lightweight binary installable globally or as devDependency. |
| Config | `memo.config.json` | Local per repo + central override. Declares relationships between repositories. |
| Credentials | `.env` | `QDRANT_URL`, `QDRANT_API_KEY`, `EMBEDDINGS_API_KEY`. Not committed. |

### Repository structure

```
memo-cli/                        ← repo: memo-cli
├── src/
│   ├── commands/
│   │   ├── write.ts             ← save decision
│   │   ├── search.ts            ← semantic search in current repo
│   │   ├── ask.ts               ← cross-repo integration query
│   │   ├── list.ts              ← chronological listing
│   │   ├── scan.ts              ← codebase archaeology
│   │   └── setup.ts             ← repo registry management
│   └── lib/
│       ├── qdrant.ts            ← Qdrant client
│       ├── embeddings.ts        ← swappable adapter
│       ├── config.ts            ← memo.config.json parser
│       └── registry.ts          ← related repo resolution
├── memo.config.json             ← local repo registry and relationships
├── package.json                 ← bin: { "memo": "./dist/index.js" }
└── .env                         ← credentials (not committed)
```

### Two knowledge layers

Memo distinguishes two types of entries in the same collection:

```
Layer 1 — Decisions (source: "agent")
  Generated when completing tasks → rationale + commit + files
  What was decided, why, discarded alternatives

Layer 2 — Product structure (source: "scan", entry_type: "integration_point")
  Generated by memo scan → API contracts, published events, patterns
  How the product is built, how to integrate with it
```

Both layers are queryable by the same commands. The `ask` command combines them to answer integration questions.

### Data model — Entry payload

| Field | Type | Description |
|---|---|---|
| `id` | uuid | Unique entry identifier |
| `repo` | string | Repository name. **Mandatory** filter on every write. |
| `org` | string | Organization or company that owns the repo. |
| `domain` | string | Business domain (e.g. backoffice, checkout, catalog). |
| `team` | string | Team responsible for the repo. |
| `story` | string | Story or task ID (e.g. USER-412). Optional for scan entries. |
| `commit` | string | Associated commit hash. Optional for scan entries. |
| `timestamp_utc` | string ISO 8601 | Date and time in UTC. Auto-generated. |
| `files_modified` | string[] | Modified or analyzed files. |
| `tags` | string[] | 2–5 categorization tags (e.g. auth, jwt, security). |
| `relates_to` | string[] | Related repos affected by this decision. |
| `rationale` | string | Full text: what was decided/how it works, why, alternatives. |
| `entry_type` | enum: `decision \| integration_point \| structure` | Entry sub-type. Allows filtering by purpose. |
| `source` | enum: `agent \| scan \| manual` | Origin. `scan` = inferred from codebase. |
| `confidence` | enum: `high \| medium \| low` | Confidence level. Relevant for `scan` entries. |

## 3. Commands

### 3.1 `memo write` — Save decision

The agent invokes this command upon completing each task or story. Generates the embedding of the rationale and saves the entry to Qdrant.

```bash
memo write \
  --repo <name> \
  --story <id> \
  --commit <hash> \
  --files <csv-paths> \
  --tags <csv-tags> \
  --rationale "<text>"
```

- The rationale must be 2–5 sentences
- Include at least one discarded alternative when applicable
- `--files` accepts relative paths separated by comma
- `timestamp_utc` is auto-generated at write time

---

### 3.2 `memo search` — Semantic search

Vector similarity search, oriented toward the current repo. First step of the agent's query protocol.

```bash
memo search "<natural query>" [--repo <n>] [--scope company] [--tags <csv>] [--limit 5]
```

- Without `--repo` uses the current repo detected from the working directory
- `--scope company` expands to all repos in the org
- Returns full rationale + metadata, ordered by similarity score

---

### 3.3 `memo ask` — Cross-repo integration query

Ecosystem-oriented query. Assumes company scope by default and combines `decision` and `integration_point` entries to answer integration questions between products.

```bash
memo ask "<integration question>" [--limit 5]
```

**Usage examples:**

```bash
memo ask "how does the payments-api API work"
memo ask "how do I integrate authentication from management-hub into my repo"
memo ask "has any team already solved rate limiting"
memo ask "where should Shopify webhooks connect in the ecosystem"
```

**Internal flow:**

```
1. Embed the query
2. Semantic search across all org repos
3. Prioritize repos related per memo.config.json
4. Combine decision + integration_point entries
5. Return structured results ordered by relevance + source repo
```

The difference from `search` is intent: `search` looks for your own decisions, `ask` looks for ecosystem knowledge to integrate with.

---

### 3.4 `memo list` — Chronological listing

Entries ordered by `timestamp_utc` descending. Useful for auditing and review.

```bash
memo list [--repo <n>] [--from <date>] [--to <date>] [--limit 20]
```

- Dates in ISO 8601 format (`YYYY-MM-DD`)
- Without filters returns the latest N entries across the org

---

### 3.5 `memo scan` — Codebase archaeology

> **Differentiating feature.** Does not exist in Memorix, Mem0, Beads, or any ADR tool on the market.

Analyzes an existing codebase, infers architectural decisions and integration points using an LLM, and bootstraps the knowledge base. Generates entries for both layers: `decision` (what was decided) and `integration_point` (how it's built, how to integrate with it).

```bash
memo scan --repo <n> --path <dir> [--depth full|module] [--target schema|code|all]
```

- Prioritizes domain, infra, config, and DB schema files
- Skips generated files, `node_modules`, and trivial tests
- Estimated cost: <$0.10 USD per mid-size repo (~200 files)
- Entries with `confidence: low` are flagged for human review
- Re-scannable without data loss — existing `scan` entries are updated

**Artifacts analyzed:**

| Artifact | LLM input | Inferred output |
|---|---|---|
| Source file | path + content | responsibility, patterns used, key dependencies |
| API / router | routes + handlers | exposed endpoints, contracts, required auth |
| Folder / module | tree + exports | purpose, boundaries, entry points |
| DB table / schema | DDL / migrations | business entity, relationships, modeling decisions |
| `package.json` | dependencies | stack decisions, chosen libraries |
| Git log | last N commits | evolution, inflection points |
| `README` / docs | full text | business context, declared architecture |

**Example of a scan-generated entry (integration_point):**

```json
{
  "repo": "payments-api",
  "entry_type": "integration_point",
  "source": "scan",
  "tags": ["api", "rest", "payments", "webhook"],
  "rationale": "Exposes POST /v1/payments to initiate transactions. Requires JWT in Authorization header issued by auth-service. Publishes payment.completed event to SNS on completion. Request schema in src/types/payment.ts. Rate limit: 100 req/min per client.",
  "files_modified": ["src/routes/payments.ts", "src/types/payment.ts"],
  "confidence": "high"
}
```

---

### 3.6 `memo setup` — Registry management

Manages `memo.config.json`, which declares the repo portfolio and its relationships.

```bash
memo setup init
memo setup add-repo
memo setup link <repo1> <repo2>
memo setup show
```

- Central config takes precedence over local in case of conflict
- Repo relationships are bidirectional by default
- Supports roles: `frontend | backend | infra | shared-lib`

## 4. Multi-Repo Model

### Single collection with mandatory repo filter

All repositories share a single Qdrant collection (`decisions`). The `repo` field is a mandatory filter on writes and the default filter on searches. This enables local and cross-repo search with the same API and no operational overhead.

### Agent query strategy

| Step | Command | When |
|---|---|---|
| 1. Local search | `memo search "<topic>" --repo <current>` | Before every non-trivial task |
| 2. Cross-repo search | `memo search "<topic>" --scope company` | When confidence in local results is low |
| 3. Integration query | `memo ask "<question>"` | When integrating with another product in the ecosystem |
| 4. Audit | `memo list --repo <x>` | Chronological review of recent decisions |

### `memo.config.json` structure

```json
{
  "org": "tf-ecommerce",
  "central_config_url": "https://raw.githubusercontent.com/org/config/main/memo.config.json",
  "repos": [
    {
      "name": "management-hub",
      "domain": "backoffice",
      "team": "platform",
      "type": "frontend",
      "relates_to": ["payments-api", "catalog-service"]
    },
    {
      "name": "payments-api",
      "domain": "checkout",
      "team": "platform",
      "type": "backend",
      "relates_to": ["management-hub", "orders-service"]
    }
  ]
}
```

### Repo relationship mechanisms

- **Shared tags:** A tag like `auth` or `jwt` across multiple repos connects related decisions in semantic search without explicit configuration.
- **`relates_to` in payload:** For decisions that directly impact multiple repos, this field makes the link explicit.
- **Registry in `memo.config.json`:** Declared relationships allow `memo ask` to know which repos to prioritize when answering integration questions.

## 5. AI Agent Integration

### Agent developer system prompt

```markdown
## Memo — Shared Decision Memory

You have access to the organization's shared knowledge base via the `memo` CLI.

### Before starting any non-trivial task
Search for prior decisions in the current repo:
  memo search "<topic>" --limit 5

If integrating with another product, ask the ecosystem:
  memo ask "how does <product> handle <topic>"
  memo ask "where is the right place to integrate <feature> in this ecosystem"

### After completing every task or story
Write the decision before finishing your response:
  memo write \
    --repo "<current-repo>" \
    --story "<story-id>" \
    --commit "<last-commit-hash>" \
    --files "<comma-separated-modified-files>" \
    --tags "<2-5-relevant-tags>" \
    --rationale "<2-5 sentences: what was decided, why, what alternatives were discarded>"

### Rules
- Always write after every completed story. No exceptions.
- Always search before starting a non-trivial task.
- Always ask before integrating with another product in the ecosystem.
- The rationale must include at least one discarded alternative when applicable.
- Before finishing any task, confirm you wrote to memo.
```

### Full operation flow

| Moment | Actor | Action |
|---|---|---|
| Task start | Developer agent | `memo search` on the task topic |
| Integrating with another product | Developer agent | `memo ask "how does X work"` |
| During task | Developer agent | Uses results to maintain architectural consistency |
| On completion | Developer agent | `memo write` with rationale and commit hash |
| Repo bootstrap | Dev / scan agent | `memo scan` to generate baseline |
| Onboarding | New dev or agent | `memo ask` to understand the ecosystem |
| Review | Tech lead | `memo list --repo <x>` to audit recent decisions |

## 6. Embeddings and Semantic Search

### How it works

An embedding converts the meaning of a text into a numerical vector. Texts with similar meaning produce nearby vectors. Memo uses embeddings exclusively as a search index — the agent always reads the `rationale` as plain text from the payload, never the vector.

```
WRITE:  rationale → embedding → Qdrant stores vector + full payload
READ:   query     → embedding → Qdrant finds similarity → returns rationale as plain text
```

### Supported providers

| Provider | Model | Cost/M tokens | Notes |
|---|---|---|---|
| OpenAI ⭐ | text-embedding-3-small | $0.02 | Recommended. ~$0.10 total for 10K entries. |
| Voyage AI | voyage-code-3 | $0.018 | Code-specialized. Better recall for technical rationale. |
| Cohere | embed-v4 | $0.016 | No meaningful advantage for this use case. |
| Ollama (local) | nomic-embed-text | Free | Requires centralized server. Viable with dedicated infra. |

The adapter is swappable with no changes to the CLI or Qdrant. Configured via `EMBEDDINGS_PROVIDER`.

### Strategy for long texts

If the rationale exceeds 512 tokens, embed a compressed summary (title + tags + main decision) and store the full text in the payload. Improves search precision without losing information.

## 7. Design Decisions

### Why Qdrant and not Mem0

Mem0 was evaluated as the primary alternative. It is the most mature agent memory platform on the market. It is not the right choice for Memo for three structural reasons:

| Dimension | Mem0 | Memo (Qdrant) |
|---|---|---|
| Rationale storage | LLM compression — may lose technical precision | Full text faithful to the agent |
| Repo relationships | Graph memory only on Pro ($249/mo) | Native `relates_to` + config registry |
| scan + ask commands | Do not exist | Memo's own differentiating features |
| Schema control | Opaque | Full control over every field |
| Cost per write | Additional LLM call | Embedding only (~$0.00001/entry) |
| Vendor lock-in | High | Low (Qdrant open source) |

### Why Memo and not Memorix

Memorix solves session continuity — local memory per agent, per project, per machine. It is complementary, not alternative. Memo solves organizational intelligence — knowledge shared across teams, repos, and agents. The `memo ask "how does payments-api work"` from `management-hub` is impossible in Memorix because projects are isolated by design.

### Why a single collection

One collection per repo would require parallel queries to N collections for cross-repo searches, complicating the client. A single collection with `repo` as a filter enables local and global search with the same API.

### Why the agent writes the rationale

The agent has complete context at the exact moment it finishes the task. The developer does not have that level of detail available post-commit. Delegating to the agent ensures capture while the context is fresh and reduces friction to zero.

### Why the system prompt and not CI/CD

By the time CI/CD runs, the decision context is already lost. The system prompt ensures capture at the precise moment — when the task is done, before the session closes.

### Write reliability

- Explicit verification instruction at the end of the prompt: *"Before finishing any task, confirm you wrote to memo."*
- The `scan` command covers the historical baseline of repos that existed before adopting Memo
- The `list` command allows auditing whether a recent period has the expected entries

## 8. Cost Estimates

| Component | Scenario | Cost | Notes |
|---|---|---|---|
| Qdrant Cloud | Free tier | $0/mo | Up to ~1GB. Sufficient for >100K entries. |
| Embeddings writes | 10K entries/year × 500 tokens | ~$0.10 total | OpenAI text-embedding-3-small |
| Embeddings search + ask | 5K queries/mo × 20 tokens | ~$0.002/mo | Queries are short |
| Repo scan | 200 files × 300 tokens | ~$0.05/scan | GPT-4o-mini. One-time per repo. |
| **Year 1 total estimate** | **5 repos, 10K entries** | **<$5 USD** | Near-zero operational cost |

2. Logic for combining `decision` + `integration_point` entries
3. Final system prompt for the developer agent
4. Integration testing with Copilot and Claude

### Phase 5 — Adoption (week 6)
1. Initial scan of all existing org repos
2. Team usage documentation
3. Central `memo.config.json` configuration
4. Full end-to-end flow validation
